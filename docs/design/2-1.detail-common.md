# 共通詳細設計書

## 1. 概要
本書は、`1.site-design.md` をもとに、アプリ共通で使用する詳細設計を整理するための文書とする。  
管理ページ・公開ページのいずれにも関わる技術構成、API共通方針、DB設計、バリデーション仕様を定義する。

## 2. 対象範囲
- 実装言語の選定
- フレームワークの選定
- データベースの選定
- API共通方針
- DB設計
- バリデーション仕様

## 3. 実装言語・技術構成
### 3.1 実装言語
- フロントエンド: TypeScript
- バックエンド: Go
- CSS: Tailwind CSS

### 3.2 フレームワーク
管理ページ
- Webフレームワーク: Svelte
- Svelteで作成し、静的HTMLをデプロイする。
- CSSフレームワーク: Tailwind CSS
- 実行形態: Go言語で作成されたAppサーバ上で管理画面および管理用APIを提供する。

公開ページ
- 実装方式: HTMLテンプレートを用いた静的HTML生成
- スタイリング: Tailwind CSS
- 配信形態: 生成済みHTML / CSS / 画像ファイルを公開する
- 補足: 公開ページは Svelte コンポーネントを実行環境に配置せず、管理ページ側の publish 処理で静的ファイルを出力する

### 3.3 データベース
- DB種別: SQLite
- 採用理由: 単一ファイルで運用しやすく、小規模ブログに適しているため

### 3.4 実行環境
- 開発環境: Svelte 開発サーバ（画面） + Go言語(API)
- 本番環境: Go言語(Appサーバ + API) + nginx（リバースプロキシ + 静的HTML配信）

### 3.5 技術選定理由
- フロントエンドはSvelte を採用し、開発効率を高める。
- サーバはGo言語で実装することにより、安定性を優先する。
- DB は SQLite とし、軽量で運用しやすい構成とする
- 管理ページで更新後に公開ページ用の静的HTMLを出力し、公開側はHTML配信のみとする

### 3.6 非機能前提
- 本番サーバはメモリ 2GB を想定する
- 他アプリ同居環境でも動作可能な軽量構成とする
- 想定アクセス数は月間 100〜1000 程度とする
- Goサーバは原則 1バイナリで運用する
- 公開ページは静的HTML配信とし、公開側のセキュリティリスクを抑える構成とする

### 3.7 環境変数
- `STATIC_EXPORT_DIR`: 公開用静的HTMLの出力先ディレクトリ
- `DATA_DIR`: データ管理用ディレクトリパス
- `TOP_PAGE_BLOG_LIMIT`: トップページに表示する新着記事件数
- `BLOGS_PER_PAGE`: 記事一覧ページの1ページあたり表示件数

補足:
- 静的HTML出力先は `STATIC_EXPORT_DIR` で指定する。デフォルト値は `./data/publish` とする
- 管理画面のビルド後 HTML は `internal/web/admin.html` を固定で使用する
- データ管理用ディレクトリのデフォルト値は `./data` とする
- 画像保存パスは `<DATA_DIR>/images/<blog_id>/<image_id>.png` とする
- `TOP_PAGE_BLOG_LIMIT` のデフォルト値は `20` とする
- `BLOGS_PER_PAGE` のデフォルト値は `20` とする

## 4. API共通設計
### 4.1 API共通方針
- リクエスト形式: 原則 JSON
- レスポンス形式: JSON
- エラー形式: JSON
- 認証制御はリバースプロキシ側で実施する

補足:
- APIは管理ページ用を中心に実装する
- 公開ページはAPIレスポンスではなく、生成済み静的HTMLを返す構成とする
- 画像アップロードAPIのみ `multipart/form-data` を使用する
- DBテーブルと同一構造の項目は、`5. DB設計` を参照する省略記法を用いる
- DBテーブルと同一構造を持たないAPIは、個別に項目を定義する
- API固有の差分項目、除外項目、配列構造のみ個別記載する
- `tabs` は DB では JSON文字列で保持するが、APIでは配列またはオブジェクトとして返却する

### 4.2 APIエラーコード方針
- 正常終了時は `200 OK` を返却する
- 新規作成成功時は `201 Created` を返却する
- バリデーションエラー時は `400 Bad Request` を返却する
- 対象データが存在しない場合は `404 Not Found` を返却する
- 同一タイトル重複などの競合時は `409 Conflict` を返却する
- サーバ内部エラー時は `500 Internal Server Error` を返却する
- エラーレスポンスの基本形式は以下とする
  - `code`: エラーコード
  - `message`: エラー内容
  - `fields`: 項目単位エラーがある場合に返却する

### 4.3 エラーコード値
- `VALIDATION_ERROR`: 入力値不正
- `NOT_FOUND`: 対象データなし
- `BLOG_TITLE_CONFLICT`: 記事タイトル重複
- `INVALID_STATUS`: `status` の値不正
- `INVALID_PUBLISHED_AT`: `published_at` の形式不正
- `INVALID_TAB_URL`: `tab_url` の形式不正
- `INVALID_IMAGE_FILE`: 画像ファイル形式不正
- `IMAGE_UPLOAD_FAILED`: 画像アップロード失敗
- `PUBLISH_TARGET_INVALID`: 静的HTML出力対象不正
- `PUBLISH_BLOG_ID_REQUIRED`: `target=blog` 時の `blog_id` 未指定
- `INTERNAL_SERVER_ERROR`: サーバ内部エラー

## 5. DB設計
### 5.1 テーブル一覧
- `site`: 共通レイアウト設定
- `blogs`: 記事データ
- `images`: 画像データ

### 5.2 テーブル詳細
#### 5.2.1 site
- テーブル名: `site`
- 想定レコード数: 1件

| カラム名 | 型 | 必須 | 説明 |
| --- | --- | --- | --- |
| id | INTEGER | 必須 | 主キー |
| site_title | TEXT | 必須 | サイトタイトル |
| site_subtitle | TEXT | 必須 | サイトサブタイトル |
| site_description | TEXT | 必須 | サイト説明 |
| tabs | TEXT | 必須 | タブ情報。JSON文字列で保持する |
| foot_information | TEXT | 必須 | フッタ情報 |
| copyright | TEXT | 必須 | コピーライト |
| updated_at | TEXT | 必須 | 更新日時 |

補足:
- `id` は主キーとする
- `site` は1レコードで運用する
- `tabs` は JSON 文字列として保存する
- `site_subtitle` はサイトタイトルの下に表示する補助文とする
- `site_description` は Markdown文字列として保存する
- `foot_information` はフッタ左側に表示する情報とする
- `updated_at` は更新時に自動更新する

#### 5.2.2 blogs
- テーブル名: `blogs`

| カラム名 | 型 | 必須 | 説明 |
| --- | --- | --- | --- |
| id | INTEGER | 必須 | 主キー |
| title | TEXT | 必須 | 記事タイトル |
| content | TEXT | 必須 | 本文 |
| summary | TEXT | 必須 | 概要 |
| category | TEXT | 任意 | カテゴリ |
| status | TEXT | 必須 | 公開状態。`public` / `private` |
| published_at | TEXT | 必須 | 公開日 |
| updated_at | TEXT | 必須 | 更新日 |

補足:
- `id` は主キーとする
- `id` は公開ページURLの識別子として使用する
- `title` は一意とする
- `status` は `public` または `private` のみ許可する
- `status` のデフォルト値は `private` とする
- `category` は 1記事につき1件とする
- `summary` は本文から自動作成する
- プロフィールページ用レコードは `title = 'about'` とする
- プロフィールページ用レコードでは `category` を使用しない

#### 5.2.3 images
- テーブル名: `images`

| カラム名 | 型 | 必須 | 説明 |
| --- | --- | --- | --- |
| id | INTEGER | 必須 | 主キー |
| blog_id | INTEGER | 必須 | 紐づけ先の記事ID |
| alt_text | TEXT | 任意 | 代替テキスト |
| created_at | TEXT | 必須 | 作成日 |
| updated_at | TEXT | 必須 | 更新日 |

補足:
- `id` は主キーとする
- `blog_id` は `blogs.id` を参照する外部キーとする
- 画像保存パスは `<DATA_DIR>/images/<blog_id>/<image_id>.png` とする
- 画像ファイル本体は DB ではなくファイルとして管理する

### 5.3 リレーション
- `blogs` 1件に対して `images` は複数件紐づけ可能とする
- `images.blog_id` により `blogs.id` と関連付ける
- プロフィールページは `blogs` の特別な1レコードとして管理する
- プロフィールページ用レコードは `title = 'about'` で識別する

### 5.4 CREATE TABLE SQL案
```sql
CREATE TABLE site (
  id INTEGER PRIMARY KEY,
  site_title TEXT NOT NULL,
  site_subtitle TEXT NOT NULL,
  site_description TEXT NOT NULL,
  tabs TEXT NOT NULL,
  foot_information TEXT NOT NULL,
  copyright TEXT NOT NULL,
  updated_at TEXT NOT NULL
);

CREATE TABLE blogs (
  id INTEGER PRIMARY KEY,
  title TEXT NOT NULL UNIQUE,
  content TEXT NOT NULL,
  summary TEXT NOT NULL,
  category TEXT,
  status TEXT NOT NULL DEFAULT 'private' CHECK (status IN ('public', 'private')),
  published_at TEXT NOT NULL,
  updated_at TEXT NOT NULL
);

CREATE TABLE images (
  id INTEGER PRIMARY KEY,
  blog_id INTEGER NOT NULL,
  alt_text TEXT,
  created_at TEXT NOT NULL,
  updated_at TEXT NOT NULL,
  FOREIGN KEY (blog_id) REFERENCES blogs(id)
);
```

### 5.5 補足
- `tabs` は JSON 文字列として保存する
- `category` は `TEXT` として保存する
- 日時は `TEXT` 型で保存し、`YYYY-MM-DD HH:MM:SS` 形式を想定する
- 静的HTMLの出力先ディレクトリは DB ではなく環境変数 `STATIC_EXPORT_DIR` で管理し、デフォルト値は `./data/publish` とする
- データ管理用ディレクトリパスはデフォルト `./data` とし、環境変数 `DATA_DIR` で変更可能とする
- 画像保存パスは `./data/images/<blog_id>/<image_id>.png` をデフォルトとする
- 管理画面専用の画像参照用URLは `/admin/images/<blog_id>/<image_id>.png` とする
- 画像保存パスおよび管理画面専用画像URLは DB には保存せず、`blog_id` と `id` から生成する派生情報とする

### 5.6 インデックス設計
#### 5.6.1 インデックス方針
- 想定アクセス数を踏まえ、初期段階では最小限のインデックスのみを作成する
- 記事一覧表示、公開状態絞り込み、公開日順表示、画像参照を主用途とする
- `title` の一意制約により生成されるインデックスを活用する
- `category` は単一値のため、通常インデックスで最適化可能とする
- カテゴリ絞り込み性能は、必要に応じて専用インデックスで対応する

#### 5.6.2 インデックス対象
- `blogs.title`
  - 用途: 記事タイトルの一意性担保、記事詳細取得
  - 備考: `UNIQUE` 制約でインデックスを兼ねる
- `blogs(status, published_at)`
  - 用途: 公開状態による絞り込みと公開日順ソート
  - 備考: 公開ページ生成時および管理画面一覧取得で使用する
- `blogs(category)`
  - 用途: カテゴリによる絞り込み
  - 備考: 記事一覧ページおよび管理画面一覧で使用する
- `images.blog_id`
  - 用途: 記事に紐づく画像一覧取得
  - 備考: 記事削除時および画像参照時に使用する

#### 5.6.3 CREATE INDEX SQL案
```sql
CREATE INDEX idx_blogs_status_published_at
  ON blogs (status, published_at);

CREATE INDEX idx_blogs_category
  ON blogs (category);

CREATE INDEX idx_images_blog_id
  ON images (blog_id);
```

#### 5.6.4 カテゴリ絞り込み性能の考え方
- `category` は単一値のため、SQLite の通常インデックスで高速化可能とする
- 現在の想定アクセス数とデータ件数では、`blogs(category)` インデックスで十分対応可能とする
- 将来的に検索条件や並び順が増えた場合は、複合インデックスの見直しを再検討する

## 6. バリデーション仕様
### 6.1 共通方針
- バリデーションは管理画面の入力時と API 受付時の両方で実施する
- 文字数上限は `5.2` のテーブル定義に記載した桁数に従う
- 必須項目が未入力の場合は保存不可とする
- 不正な入力がある場合は項目単位でエラーメッセージを表示する
- API はバリデーションエラー時にエラー JSON を返却する

### 6.2 サイト設定
- `site_title`
  - 必須
  - 100 文字以内
- `site_description`
  - 必須
  - 1000 文字以内
  - Markdown 文字列として入力可能
- `tabs`
  - 必須
  - 最大 10 件
- `tabs[].tab_label`
  - 必須
  - 20 文字以内
- `tabs[].tab_url`
  - 必須
  - 100 文字以内
  - `/` から始まる内部 URL のみ許可する
- `copyright`
  - 必須
  - 100 文字以内

### 6.3 記事
- `title`
  - 必須
  - 100 文字以内
  - 重複不可
- `content`
  - 必須
  - 20,000 文字以内
  - Markdown 文字列として入力可能
- `summary`
  - 入力不可
  - `content` から自動生成する
- `category`
  - 任意
  - 20 文字以内
  - 公開URL `/blogs/category/{category}` に使用できない文字を含む場合は保存不可
- `status`
  - 必須
  - `public` または `private` のみ許可する
  - 未指定時は `private`
- `published_at`
  - 必須
  - `YYYY-MM-DD HH:MM:SS` 形式
- `updated_at`
  - 入力不可
  - 更新時に自動設定する
- プロフィール編集時
  - `title` は `about` 固定
  - `category` は使用しない

### 6.4 画像アップロード
- `blog_id`
  - 必須
  - 対応する記事が存在すること
- `file`
  - 必須
  - 画像ファイルのみ許可する
  - 保存時に最大 `1920x1080` へリサイズする
- `alt_text`
  - 任意
  - 200 文字以内

### 6.5 補足
- バリデーション仕様の詳細なエラーメッセージ文言は実装時に定義する
- URL 使用不可文字の具体一覧は、実装時に使用ライブラリまたは共通関数に合わせて定義する

## 7. カテゴリ単数仕様
- `category` は 1記事につき1件のみ設定可能とする
- 複数カテゴリの指定は不可とする
- カテゴリ別記事一覧は静的HTMLとして事前生成する
- `blogs` テーブルでは `category TEXT` として保持する
- API の入出力項目も `category` を使用する
